# Architecture Practicies
## Фаза Discovery
![](schemas/sa_activities.svg)

Сама фаза Discovery может быть разбита не 3 этапа:
- [Подготовка](#подготовка) (on boarding)\
<small>10% времени</small>
- [Интенсивный этап](#интенсивный-этап) (workshop-ы, коммуникации)\
<small>30% времени</small>
- [Финализация](#финализация) (результаты)\
<small>60% времени</small>

Вся фаза Discovery длится от пары недель до пары месяцев.

### Подготовка
#### Тактика
- Проанализировать информацию из RFP.
  - RFP **не являются** источником конечной истины
- Организовать встречи с ключевыми участниками RFP с технической точки зрения
- Проверить gap-ы, предположения и предложенные технические решения из RFP
- Предложить резонные изменения, если они нужны (но быть аккуратными при коммуникации)
- Сделать матрицу RACI, определить активности

### Интенсивный этап
Чтобы workshop-ы с заказчиком прошли хорошо (особенно, если несколько дней в офисе заказчика надо проводить), нужно четко спланировать встречи
- Время
- Агенда
- Требуемые участники
- Input
- Output

#### Ответственность членов команды
- Координатор
  - Формирование команды для Discovery
  - Представление команды заказчику
  - RACI
- Project (Delivery) Manager
  - Scope
  - Project Timeline, Roadmap
- Бизнес-аналитик
  - Организационная структура, stakeholder-ы
  - Features, уточнение по FR
  - Виденье
- Архитектор решений
  - Техническое решение, SAD
  - уточнение ASR
  - Стек технологий
  - Оценки

#### Каналы коммуникации
- Архитектору решений стоит принимать участик только в мероприятиях, относящихся к его задачам
- Определить stakeholder-ов проекта по части техники и взаимодействовать с ними
- Оставить FR-ы бизнес аналитику и только контролировать их сбор
- Сконцентрироваться на ARS и техническом решении
- Собирать domain/technical knowledge

#### Тактика
- Уточнение scope-а
- Составление архитектурного backlog-а для workshop-ов
- Проводить дискуссии вокруг набросков/эскизов
- Не пытаться финализировать все на свете за один раз

##### Quality Attribute Workshop
1. Презентация и введение в QAW
2. Презентация бизнесса/миссии
3. Презентация архитектурного плана
4. Определение архитектурных драйверов среди stakeholder-ов
5. Scenario Brainshtorm
6. Консолидация сценариев
7. Приоретизация сценариев
8. Уточнение сценариев

#### Результат
- Список ASR составлен и предварительно обговорен со stakeholder-ами
- Готовы эскизы диаграм важных частей системы
- Большинство предположений уточнены и решены
- Технологический стек предварительно обсужден

### Финализация
#### Цели
- Определить и завершить основные [результаты работы](#результаты-работы)
- Выровнется со stakeholder-ами по предлагаемому решению
- Сформировать подходящую команду, которая может реализовать решение

#### Результаты работы
- Принято участие в формирование
  - Виденье проекта
  - Backlog, WBS, Roadmap
- Завершено
  - Solution Architecture Document
    - Подходящие представления
    - Согласованные ASR-ы
    - Аттрибуты качества
  - Полностью определен технологический стек
  - Подготовлен PoC для критических частей системы

#### Тактика
- Не предполагать и не ожидать. Уточнять, анализировать, проектировать
- Сделать backlog для финализации системного дизайна
- Организовать регулярные созвоны для проверки статуса с внутренними и внешними stakeholder-ами
- Разделить время:
  - 25% на коммуникацию
  - 75% на проектирование и PoC
- Отслеживать свой прогресс
- Готовить артефакты дизайна итеративно
- Не кодить слишком много!

#### Формирование команды
- Подбор команды, которая имеет опыт в нужных технологиях
- Интеврью тех.лида
- Интервью экспертов в выбранных технологиях
- Включение их в PoC
